# Staging deploy (API)

Host the NestJS API on Render (or similar) so the mobile app works without your PC on the LAN.

## Prerequisites

- [Neon](https://neon.tech) Postgres — run all migrations in `db/migrations/` (including `003_sync_stats.sql`)
- [Upstash](https://upstash.com) Redis — for BullMQ email sync + notifications
- Google OAuth client (same as local) — add staging redirect URIs if needed
- [Render](https://render.com) account (free tier works for staging)

## Option A — Render Blueprint (`render.yaml`)

1. Push repo to GitHub.
2. Render Dashboard → **New** → **Blueprint** → connect repo.
3. Set secret env vars when prompted:

| Variable | Source |
|----------|--------|
| `DATABASE_URL` | Neon connection string |
| `REDIS_URL` | Upstash `rediss://…` URL |
| `ENCRYPTION_MASTER_KEY` | Same 64-char hex as local (or new — invalidates encrypted tokens) |
| `GOOGLE_CLIENT_ID` | Google Console |
| `GOOGLE_CLIENT_SECRET` | Google Console |
| `PLAID_CLIENT_ID` | [Plaid dashboard](https://dashboard.plaid.com/) (optional — refund radar) |
| `PLAID_SECRET` | Plaid **sandbox** secret |
| `PLAID_ENV` | `sandbox` |

`JWT_SECRET` can be auto-generated by Render.

See [PLAID_SETUP.md](./PLAID_SETUP.md) for where keys go (API only — not mobile).


**Staging auth (Expo Go):** The app stores a JWT on your phone. If you switch from local API to Render, you get `401 Invalid token` because Render’s `JWT_SECRET` differs from local. Fix either:

1. **Sync secret (keeps same session):** Copy `JWT_SECRET` from `apps/api/.env` into Render → Environment → override the auto-generated value → redeploy.
2. **Reset session (after `ALLOW_DEV_AUTH=true` on Render):** In the app → Settings → **Reset session**, or pull to refresh on the dashboard (app auto-clears bad tokens).

Set `ALLOW_DEV_AUTH=true` on Render so staging can issue dev login tokens (not for public production).

**Generate a new shared secret** (local + Render must match):

```cmd
python scripts\generate_jwt_secret.py
```

Update `JWT_SECRET` in `apps/api/.env` and Render, then **Reset session** in the mobile app.

4. Deploy. Note the URL, e.g. `https://returnrider-api.onrender.com`.

### If deploy still uses an old build command

Render Blueprints **do not always auto-update** build settings on an existing service. After pushing an updated `render.yaml`, either:

**Option A — Sync Blueprint:** Render Dashboard → **Blueprints** → `returnrider-api` → **Manual sync** (or re-sync from repo).

**Option B — Edit the web service directly:** `returnrider-api` → **Settings** → **Build & Deploy**:

| Field | Value |
|-------|--------|
| Root Directory | `.` (repo root) |
| Build Command | `NPM_CONFIG_PRODUCTION=false npm ci && npm run build -w @returnrider/api` |
| Start Command | `npm run start -w @returnrider/api` |

Add env var `NODE_VERSION` = `20` if the service still picks Node 24.

The API `build` script also uses `npx` to fetch Nest/TypeScript when devDependencies are omitted (production `npm install`).

## Option B — Manual Render Web Service

- **Root directory:** `apps/api`
- **Build:** `npm install && npx prisma generate && npm run build`
- **Start:** `npm run start`
- **Health check path:** `/health`

## After deploy

### Verify

```text
GET https://YOUR-API.onrender.com/health
→ { "status": "ok", "service": "returnrider-api", "features": { "plaid": true, "gmail": true } }

GET https://YOUR-API.onrender.com/api/docs
→ Swagger UI
```

If `"plaid": false`, add `PLAID_*` on Render and redeploy (local `apps/api/.env` does not affect staging).


### Mobile `.env`

```env
EXPO_PUBLIC_API_URL=https://YOUR-API.onrender.com/api/v1
```

Restart Expo: `npx expo start -c`

Install mobile deps (mobile is **not** an npm workspace — run from its folder):

```cmd
cd apps\mobile
npm install
```

Or from repo root: `npm run mobile:install`


### Google OAuth (physical phone)

Expo Go still uses `https://auth.expo.io/@ahmed5145/returnrider` — no change unless you switch to a dev build.

### CORS

API uses `origin: true` (all origins) — fine for staging. Tighten with `CORS_ORIGINS` before public launch.

### Email sync scheduler

Runs automatically every 15 min via BullMQ. Disable with `EMAIL_SYNC_SCHEDULER_ENABLED=false` if debugging.

### Keep the API awake (free tier)

Render **free** web services spin down after ~15 minutes idle. While asleep, BullMQ sync and scheduled push jobs do not run.

**Recommended — UptimeRobot (free, no credit card):**

1. Sign up at [uptimerobot.com](https://uptimerobot.com) (free tier: 50 monitors).
2. **Add New Monitor**
   - Monitor type: **HTTP(s)**
   - Friendly name: `ReturnRider API`
   - URL: `https://returnrider-api.onrender.com/health`
   - Monitoring interval: **5 minutes**
3. **Create Monitor**
4. Within ~30 minutes the API should stay warm. Confirm in Render → `returnrider-api` → **Logs** (requests every 5 min).

**Verify it works:** Leave the mobile app closed for 25+ minutes, then open the dashboard. The “Inbox sync is behind” banner should **not** appear.

**Do not add payment to Render** just for keep-warm — Render **cron jobs** require a card on file. We intentionally removed cron from `render.yaml` for staging.

**Later (paid):** Upgrade `returnrider-api` to **Starter** (~$7/mo) for always-on without an external pinger.

## Local vs staging

| | Local | Staging |
|---|-------|---------|
| API URL | `http://LAN_IP:3000/api/v1` | `https://….onrender.com/api/v1` |
| PC must be on | Yes | No |
| Free tier sleep | No | Render free spins down after idle |

## Troubleshooting

| Issue | Fix |
|-------|-----|
| `nest: not found` on Render | `NODE_ENV=production` skips devDeps — use `NPM_CONFIG_PRODUCTION=false` in build (see `render.yaml`) |
| Build fails on Prisma | Ensure `DATABASE_URL` set; `postinstall` runs `prisma generate` |
| 502 on wake | Free tier cold start — wait 30–60s |
| `Network request timed out` on phone | Render slept — open `/health` in Safari first, then pull to refresh in app. App now retries once with a longer timeout. |
| Timeouts keep happening | Confirm `apps/mobile/.env` uses `https://returnrider-api.onrender.com/api/v1` (not `localhost` or old LAN IP) |
| Gmail token exchange fails | `GOOGLE_CLIENT_ID` / `SECRET` match mobile client |
| Sync not running | Check `REDIS_URL`; Upstash must allow BullMQ connections |
